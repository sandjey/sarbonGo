# SarbonGo — обязательные практики

## Базовые заголовки (клиент → сервер)

- Для эндпоинтов, где это предусмотрено middleware (`RequireBaseHeaders` и аналоги), клиент **обязан** передавать:
  - **`X-Device-Type`**: `web` | `ios` | `android`
  - **`X-Language`**: `ru` | `uz` | `en` | `tr` | `zh` (ровно **5** языков)
  - **`X-Client-Token`**: токен клиентского приложения
- Для защищённых маршрутов дополнительно — **`X-User-Token`** (и прочие заголовки по роли), если это уже заложено в коде.
- Новые публичные/полу-публичные ручки не ослабляют проверку базовых заголовков без явной причины и согласования.

## Ответы и язык

- Тексты для пользователя в ответах — через **ключи i18n** и **`resp.*Lang`** (или принятый в проекте слой), с опорой на **`X-Language`**.
- Не хардкодить пользовательские строки на одном языке в handler’ах без необходимости; новые сообщения — добавлять во **все 5** локалей в `internal/server/resp/i18n.go` (или актуальный файл i18n).
- Поле **`description`** (и аналоги) в JSON-ответах, если генерируется с переводом, должно соответствовать выбранному языку.

## OpenAPI (`docs/openapi.yaml`) = код

- При **любом** изменении HTTP API (путь, метод, параметры, тело, коды ответов, схема данных) — **в том же PR/задаче** обновлять **`docs/openapi.yaml`**, чтобы спецификация совпадала с реализацией.
- Теги, `security`, описания ошибок и примеры — поддерживать в актуальном виде; не оставлять в Swagger удалённые или переименованные маршруты без явной пометки `deprecated`, если старый путь ещё жив в коде.

## Процесс работы агента (перед началом правок)

1. **Проанализировать** запрос: цель, затронутые модули, риски, обратная совместимость.
2. Если задача **неясна на 100%** (объём, поведение по краям, совместимость с клиентами, формат API) — **сначала задать уточняющие вопросы** пользователю.
3. **Начинать** реализацию после того, как требования подтверждены или однозначно выведены из кода/спеки.

Краткость правила не должна ослаблять эти пункты: при сомнении — спросить, потом делать.
